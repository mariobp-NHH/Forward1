What is a Flask form (WTForms)?

Web forms, such as text fields and text areas, give users the ability to send data to your application to use it to perform an action, or to send larger areas of text to the application. In our app, we are using forms to register, to login, and to pass the number of kilometres, the means of conveyance and the type of fuel. In particular, we will use WTForms that is a flexible forms validation and rendering library for Python web development.

In our app, we are going to create a registration form. In that registration form, the users must introduce their username, their email, and password. Given that all the users must have these three fields, it is a good idea to create a class. Once that a single user introduces its username, email, and password, we will have an object generated by that class. Moreover, the classes that we are going to create can use methods programmed in flask that allows us to detect if the emails are incorrect, if the usernames do not fulfil certain requirements, etcetera.

Session and secret key

If you have set Flask.secret_key (or configured it from SECRET_KEY) you can use sessions in Flask applications. A session makes it possible to remember information from one request to another. The way Flask does this is by using a signed cookie. The user can look at the session contents but can't modify it unless they know the secret key, so make sure to set that to something complex and unguessable.

The secret key is useful to protect our app from Cross Site Request Forgery attacks. Cross-Site Request Forgery (CSRF) is an attack that forces authenticated users to submit a request to a Web application against which they are currently authenticated. CSRF attacks exploit the trust a Web application has in an authenticated user. (Conversely, cross-site scripting (XSS) attacks exploit the trust a user has in a particular Web application). A CSRF attack exploits a vulnerability in a Web application if it cannot differentiate between a request generated by an individual user and a request generated by a user without their consent.

How to create a random secret key

  1. Open a terminal in a Visual Studio.

  2. In the Visual Studio terminal, type python and then you will have tree arrows indicating that you are in a Python terminal >>>

  3. In the Python terminal, type: import os

  4. Type again: os.urandom(24).hex()

  5. You will obtain a random number that can be your secret key: application.config['SECRET_KEY'] = 'your secret key' We need to place that line of code in the __init__.py file where we define our flask application.

  6. You can close the Python terminal by typing quit().

  7. To protect our secret key, when we deploy our app in AWS, we will protect the key by using an environmental variable.

Creating forms

To create WTForms, we proceed in three different steps:

  1. We need to install some packages to use WTForms.
    1. pip install flask-wtf
    2. pip install email-validator

  2. We need to import some packages and some validators.

  3. We create a class named RegistrationForm with four fields and submit button. For each field, we create a label, and we introduce some validators to guarantee that the user introduces the right type of information. In the message flashing section, we will flash some messages to the user if the information introduced does not satisfy the validators requirements.

Creating a form

from flask_wtf import FlaskForm
from wtforms import StringField, PasswordField, SubmitField
from wtforms.validators import DataRequired, Length, Email, EqualTo   

class RegistrationForm(FlaskForm):
    username = StringField('Username', 
                    validators=[DataRequired(), Length(min=2, max=30)])
    email = StringField('Email',
                    validators=[DataRequired(), Email()])
    password = PasswordField('Password', validators=[DataRequired()])
    confirm_password = PasswordField('Confirm Password',
                    validators=[DataRequired(), EqualTo('password')])
    submit = SubmitField('Sign Up')

Validating forms

To validate WTForms, we proceed in three different steps:

  1. We need to import some modules from Flask that we will use to validate our WTForm. Those modules are not strictly necessary, and if you use a different way to validate your WTForm, you will not need them.

  1. We need to import the class RegistrationForm that we have created previously. That class is located in the folder capp, the package users in the forms.py file.

  2. When we define the route, we need to follow four steps:
    1. We need to create a route for the registration form. That route not only accept GET requests, but also POST requests since the user will introduce some data in the form.
    2. In the function register(), we need to define the form= RegistrationForm().
    3. If the user does not validate the form, we will disclose a register.html file where the user introduces her information. We define that register.html file in the next section.
    4. If the user validates the form by clicking the submit button, we redirect the user to the home page.

Validating a form

from flask import render_template, Blueprint, redirect, flash, url_for
from capp.users.forms import RegistrationForm

users=Blueprint('users',__name__)

@users.route('/register', methods=['GET','POST'])
def register():
    form = RegistrationForm()
    if form.validate_on_submit():
        return redirect(url_for('home.home_home'))
    return render_template('users/register.html', title='register', form=form)

Message Flashing.

We are going to disclose two types of messages. First, in the register.html file, we disclose messages when the user introduces information that do not satisfy the validators requirements introduced in the forms.py file. Second, in the home.html file we disclose a message when the user registers her information correctly.

Messages in the register.html file. By using Jinja, we introduce and if-else statement that helps us to manage the error messages.

If the user introduces some information in the username field that violates the validation requirements introduced in the forms.py file, we collect those errors and disclose them inside a container with the bootstrap class “invalid-feedback.” Otherwise, we do not disclose any error message.

Messages in the home.html file. By using Jinja, we introduce a with, an if and for statement in our home.html file to disclose information to the user when she registers in the app.

In those statements, we take the flashed messages from the route, and we disclose them taken into account the category of the message and creating a specific bootstrap depending on the category of the message.

Messages in the register.html file

<div class="form-group">
    {{ form.username.label(class="form-control-label") }}
        {% if form.username.errors %}
            {{ form.username(class="form-control form-control-lg is-invalid") }}
            <div class="invalid-feedback">
                {% for error in form.username.errors %}
                    <span>{{ error }}</span>
                {% endfor %}
            </div>
        {% else %}
            {{ form.username(class="form-control form-control-lg") }}
        {% endif %}
</div>
Messages in the home.html file

{% with messages = get_flashed_messages(with_categories=true) %}
    {% if messages %}
        {% for category, message in messages %}
            <div class="alert alert-{{ category }}">
                {{ message }}
            </div> 
        {% endfor %}
    {% endif %}
{% endwith %}

To deploy our app in AWS we need to consider two things.

  1. In the requirements.txt file, we need to add all the packages that we have used in this chapter, so the server can install the same packages.

  2. We need to protect our secret key by using instrumental variables. Please watch the video in which I show the changes that must be introduced in the __init__.py file before to place that file in GitHub and the changes that we need to introduce in AWS to create an environmental variable to protect our secret key.

What are environment variables? Environment variables are predetermined values that are typically used to provide the ability to configure a value in your code from outside of your application.

How can I keep these files secure? For instance, AWS keys are a valuable source. People run bots with the sole purpose of trying to scan Github for keys. If someone finds an AWS key, they could use this key to access resources such as running a bitcoin operation at your expense. This isn't to scare you, its to make you aware so you avoid your keys getting compromised.
